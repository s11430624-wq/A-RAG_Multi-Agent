# A-RAG × Multi-Agent 人工評分標準 (Manual Review Rubric)

本標準定義實驗最終產出程式碼之人工評估細則。為確保評分之客觀性、避免主觀偏見，所有評審必須採用 **雙盲審查（Blind Review）**：評審人員在評分時，僅能看到 `task_id`、題目需求、以及最終產出的原始碼（Artifacts），**絕對無法得知該程式碼是由 Strategy A, C 還是 E 產出**。

---

## 一、 需求達成度得分 (Requirement Score, 0–2)

本指標評估產出程式碼是否完整滿足了題目所要求的功能性與邏輯規範。

| 分數 | 等級 | 明確判定標準 |
| :---: | :---: | :--- |
| **2** | **完整達成 (Full)** | 程式碼完全符合題目的所有功能性需求與邊界條件，沒有漏掉任何細節，邏輯正確無瑕疵。 |
| **1** | **部分達成 (Partial)**| 程式碼實作了核心邏輯，但遺漏了次要需求或未妥善處理邊界情況（例如：未正確處理 0 或大於 100 的異常值；或未按照政策對及格率進行 float 四捨五入等）。 |
| **0** | **未達成 (Failed)** | 程式碼邏輯有重大錯誤、完全無法運行、或遺漏核心功能需求。 |

---

## 二、 程式碼品質得分 (Quality Score, 1–5)

本指標依據專案 `STYLE_GUIDE.md` 與一般 Python 工程最佳實踐，評估程式碼的可讀性、可維護性、架構設計與例外處理。

| 分數 | 等級 | 明確判定標準 |
| :---: | :---: | :--- |
| **5** | **卓越 (Excellent)** | - 完全符合 `STYLE_GUIDE.md` 命名與註解標準。<br>- 例外處理（Exception Handling）極為嚴謹，無潛在崩潰點。<br>- 結構優美、無重複代碼、使用 Pythonic 寫法。<br>- 模組化與可讀性極高。 |
| **4** | **良好 (Good)** | - 符合絕大部分的 `STYLE_GUIDE.md` 規範。<br>- 具有基本的異常處理，邏輯清晰。<br>- 結構合理，可維護性良好，但有些微可改善的冗餘寫法。 |
| **3** | **合格 (Acceptable)**| - 能運作且結構可接受。<br>- 缺少足夠的註解，或命名不夠精準（但未違反 Style Guide 的強制規範）。<br>- 異常處理較為粗糙，主要使用寬泛的 `except Exception:`。 |
| **2** | **欠佳 (Poor)** | - 程式碼混亂、難以閱讀。<br>- 違反 `STYLE_GUIDE.md` 中的明確命名或排版規範。<br>- 存在大量重複或冗餘程式碼（Copy-paste）。<br>- 完全沒有異常處理。 |
| **1** | **極差 (Critical)** | - 程式碼呈現「義大利麵條式（Spaghetti Code）」混亂。<br>- 存在明顯的語法錯誤或根本無法執行的壞味道。<br>- 完全不具備任何可維護性。 |

---

## 三、 API 正確性 (API Correctness, 0 / 1)

評估產出程式碼中所調用、進口（Import）之既有系統 API 是否正確。

- **`1` (正確 - Pass)**:
  - 程式碼中所調用的所有外部模組、類別、函式及傳參，皆完全存在於系統既有的程式庫中（如 `student_system` 下 `student.py`、`course.py`、`grade.py`、`utils.py` 實際定義者，或 `API_SPEC.md` 所載）。
- **`0` (錯誤 - Fail)**:
  - 程式碼中只要調用了**任何一個**實際上不存在、或參數定義完全不符的既有系統 API。

---

## 四、 API 幻覺 (Hallucinated API, 0 / 1)

評估模型是否憑空捏造、虛構了系統中不存在的函式、屬性或第三方套件。

- **`1` (有幻覺 - Hallucinated)**:
  - 模型在不確定的情況下，憑空捏造了系統根本沒有定義的類別方法（例如在 `student` 物件上虛構了並不存在的 `.get_all_failed_courses_by_semester()`），或引入了未安裝的第三方函式庫。
- **`0` (無幻覺 - Clean)**:
  - 程式碼中完全沒有任何憑空虛構的 API 或是憑空捏造的物件屬性。

---

## 五、 雙盲審查執行流程 (Blind Review Process)

1. **數據脫敏**：由系統管理員將 `results/raw/` 內的 JSONL 輸出進行去標識化處理，移除其中的 `strategy` 欄位，僅保留 `task_id`、`run_id` 及其產出的程式碼檔，並重新隨機編號。
2. **獨立評分**：交由至少兩位不參與 Agent 開發的評審人員，依照本 Rubric 進行獨立盲審評分。
3. **分歧仲裁**：若兩位評審評定的 `requirement_score` 差距大於等於 1 分，或 `quality_score` 差距大於等於 2 分，應由第三位評審（或實驗主持人）介入進行合議仲裁，確定最終得分。
